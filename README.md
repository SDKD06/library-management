# 📚 Library Management System

A full-stack Library Management System with an **Angular** frontend, a **Spring Boot (Java)** backend, and a **MySQL** database. It handles the everyday operations of a library — managing books, members, and borrow/return transactions — secured with JWT authentication and role-based access.

Beyond standard CRUD, it includes a **Book Demand Predictor**: a feature that forecasts which titles are most likely to be borrowed next, based on recent borrowing trends and genre popularity, complete with an interactive demand-forecast chart.

---

## ✨ Features

- **Authentication & Authorization** — JWT-based login with role-based access (`ADMIN`, `LIBRARIAN`).
- **Books** — Add, edit, delete, and list books (title, author, genre).
- **Members** — Manage library members and their status.
- **Transactions** — Borrow and return books, with live status tracking (`ACTIVE`, `COMPLETED`, `OVERDUE`).
- **Automatic Overdue Detection** — A scheduled job flags loans as `OVERDUE` once they pass the 14-day limit (runs at startup and daily at midnight).
- **Dashboard** — At-a-glance stats: total books, available, borrowed, overdue, members, and a genre breakdown.
- **Reports** — Most borrowed books, most active members, and overdue summaries.
- **Book Demand Predictor** ⭐ — Forecasts the next most-likely-to-be-borrowed books, with summary cards, a ranked list, and a genre-colored bar chart.

## 🚀 Getting Started

### Prerequisites

- **Java 17+** (JDK)
- **Node.js 18+** and **Angular CLI** (`npm install -g @angular/cli`)
- **MySQL 8** running locally

### 1. Database setup

Create the database (the schema is auto-generated by Hibernate on first run):

```sql
CREATE DATABASE bookdb;
```

### 2. Configure the backend

Open `backend/src/main/resources/application.properties` and set your own MySQL credentials:

```properties
spring.datasource.url=jdbc:mysql://localhost:3306/bookdb
spring.datasource.username=YOUR_DB_USERNAME
spring.datasource.password=YOUR_DB_PASSWORD
spring.jpa.hibernate.ddl-auto=update

# JWT secret (use your own long, random string)
jwt.secret=YOUR_SECRET_KEY
```

> **Note:** `application.properties` is git-ignored so credentials are never committed. Each developer sets their own.

### 3. Run the backend

From the `backend/` folder:

```bash
# Windows (PowerShell)
.\mvnw spring-boot:run

# macOS / Linux
./mvnw spring-boot:run
```

The API starts on **http://localhost:8080**.

### 4. Run the frontend

From the `frontend/` folder:

```bash
npm install
ng serve
```

The app starts on **http://localhost:4200**.

## 🔑 Logging In

Users are stored in the `users` table with BCrypt-hashed passwords and a role (`ADMIN` or `LIBRARIAN`). Register a user through the auth endpoint (or seed one directly in the database), then log in via the UI at `http://localhost:4200`.

## 📡 API Overview

All endpoints are served from `http://localhost:8080`. Protected routes require a `Bearer <token>` header, which the Angular interceptor attaches automatically after login.

| Method | Endpoint              | Description                               |
| ------ | --------------------- | ----------------------------------------- |
| POST   | `/auth/register`      | Register a new user                       |
| POST   | `/auth/login`         | Log in and receive a JWT                  |
| GET    | `/books`              | List all books                            |
| POST   | `/books`              | Add a book                                |
| PUT    | `/books/{id}`         | Update a book                             |
| DELETE | `/books/{id}`         | Delete a book (ADMIN only)                |
| GET    | `/members`            | List members                              |
| GET    | `/transactions`       | List transactions                         |
| POST   | `/transactions`       | Borrow a book                             |
| GET    | `/predictions?limit=` | Top predicted books overall               |
| POST   | `/predictions`        | Predictions biased toward selected genres |

> Exact sub-routes are defined in the controllers under `backend/.../controller/`.

## 🔮 How the Book Demand Predictor Works

The predictor estimates which books are most likely to be borrowed next. It is a **transparent, recency-weighted scoring model** (no machine learning) that runs over the existing transaction history.

Each book gets a score out of 100, blending three signals:

| Signal             | Weight | What it measures                                     |
| ------------------ | ------ | ---------------------------------------------------- |
| **Book momentum**  | 45%    | How much _this book_ has been borrowed recently      |
| **Genre momentum** | 35%    | How popular _its genre_ is right now                 |
| **Genre match**    | 20%    | A bonus if the book matches the genre you focused on |

**Recency matters most.** Every borrow is weighted so that recent activity counts more than old activity — a borrow's weight halves every 30 days. So a book borrowed last week pushes its score up far more than one borrowed months ago.

The predictor page shows:

- **Summary cards** — top predicted pick, hottest genre, and how many are available.
- **Demand Forecast chart** — a bar chart of predicted scores, colored by genre.
- **Ranked list** — each book with its score, availability, and a plain-English reason (e.g., _"3 borrows in the last 30 days; Technology is trending"_).

You can also focus on specific genres to bias the forecast toward what you care about.
